# trade_flow 문서 운영 규칙

## 1. 문서 구성과 우선순위

| 우선순위 | 문서 | 역할 |
|---------:|------|------|
| 1 | [`specs/2026-07-21-trade-flow-tech-spec.md`](specs/2026-07-21-trade-flow-tech-spec.md) | 구현, 테스트, 운영 승인 기준 |
| 2 | [`specs/2026-07-21-trade-flow-design.md`](specs/2026-07-21-trade-flow-design.md) | 배경, 목표, 전략 선택 이유 |
| 3 | 외부 자료 | 아이디어와 사실 확인을 위한 참고 자료 |

- Tech Spec과 설계 문서가 충돌하면 Tech Spec을 따른다.
- 설계 문서는 결정 이유와 핵심값을 요약할 수 있지만 구현은 Tech Spec의 정확한 계약을 따른다.
- Notion, API 문서, 대화 기록은 Tech Spec에 반영되기 전까지 구현 기준이 아니다.
- 동일한 목적의 `final`, `proposal`, `copy` 문서를 다른 폴더에 만들지 않는다.

## 2. 문서 상태

| 상태 | 의미 | 사용 범위 |
|------|------|-----------|
| Draft | 작성 중이며 결정되지 않은 내용이 있다. | 구현 기준으로 사용하지 않는다. |
| In Review | 검토 중이며 열린 질문의 결정이 필요하다. | 승인된 절만 사전 작업에 사용한다. |
| Approved — Phase N | 해당 Phase 구현과 검증에 필요한 내용이 승인되었다. | 표시된 Phase의 기준으로 사용한다. |
| Superseded | 다른 문서로 대체되었다. | 구현에 사용하지 않는다. |

Phase 승인은 이후 Phase의 실계좌 정책까지 자동 승인하지 않는다. Phase 1 승인은 백테스트 구현과 평가 기준만 승인한다.

## 3. 변경 절차

1. 변경이 전략 정책, 실행 정책, 문서 정리 중 어디에 해당하는지 식별한다.
2. 수익률·회전율·MDD에 영향을 주는 전략 변경은 Phase 1 재검증 대상으로 표시한다.
3. 주문 수량·체결·멱등성·계좌 보호에 영향을 주는 변경은 Phase 2 재검증 대상으로 표시한다.
4. 정책 수치와 도메인 정의의 원장은 Tech Spec Section 0이다.
5. Tech Spec 본문은 Section 0의 정책을 참조하고 같은 수치를 다시 정의하지 않는다.
6. 열린 질문이 해결되면 Section 8에서 제거하고 결정 결과를 해당 본문에 흡수한다.
7. 변경 이력은 Section 9에 80자 이내 한 줄로 기록한다.
8. 설계 문서가 같은 정책을 요약하고 있으면 의미가 어긋나지 않도록 함께 수정한다.
9. 변경 후 자기 검증과 두 문서의 교차 검증을 수행한다.

## 4. 사용자 승인이 필요한 변경

다음 변경은 백테스트 결과 또는 구현 편의만으로 자동 확정하지 않는다.

- 실계좌 투입 자본과 자본 상한
- 손절·일일 손실 한도·종목별 비중 상한
- 위험 레짐의 실제 주문 정책
- 감정점수를 주문에 반영하는 결정
- 공매도, 레버리지, 자동 환전의 범위 추가
- Phase 전환과 실계좌 활성화

## 5. 자기 검증

문서 변경자는 다음 항목을 확인한다.

- 입력, 출력, 상태 변화, 정상 흐름, 예외 흐름이 정의되어 있다.
- 동일 입력과 설정으로 전략 결과를 재현할 수 있다.
- 미래 참조, 생존 편향, 거래비용을 서로 다른 검증 항목으로 다룬다.
- 주문 응답 유실, 부분 체결, 취소 실패, 재실행 시 동작이 정의되어 있다.
- 미확정 외부 API 사실을 확정된 정책처럼 작성하지 않는다.
- 보안, 로그, 백업, 알림 실패가 누락되지 않는다.
- 해소된 질문과 폐기된 정책이 본문에 누적되어 있지 않다.
- 구현자가 `docs/superpowers` 밖의 문서를 읽지 않아도 작업을 시작할 수 있다.

## 6. 구현 결과 기록

- 구현 상세와 일회성 작업 절차는 태스크 파일 또는 PR description에 기록한다.
- 실제 승인된 설정은 실행 manifest의 `config_hash`, `data_hash`, `universe_hash`로 보존한다.
- 구현 결과를 과거형으로 Tech Spec에 누적하지 않는다.
- 반복적으로 필요한 운영 지식만 Tech Spec의 계약이나 불변식으로 반영한다.

# trade_flow 설계 문서

날짜: 2026-07-21
상태: 설계 확정 (구현 계획 수립 전)

## 1. 개요

한국투자증권(KIS) Open API 기반 **미국주식 일봉 스윙 자동매매 프로그램**.
미국 시장으로 시작하고, 검증 완료 후 한국 시장으로 확장한다.

핵심 원칙:

- **전략 = 순수 함수.** 백테스터와 라이브 실행기가 동일한 전략 코드를 호출한다. 검증한 코드가 그대로 실전에 돈다.
- **스케줄 실행형.** 상주 데몬·웹소켓 없음. 하루 1회 실행 후 종료.
- **fail-safe.** 상태가 불확실하면 거래하지 않고 사람을 호출한다.

## 2. 확정된 결정 사항

| 항목 | 결정 | 비고 |
|---|---|---|
| 증권사 | 한국투자증권 KIS Developers | REST API 하나로 국내+미국 지원, 모의투자 서버 제공 |
| 언어/환경 | Python, 로컬(맥)/서버 상주 스케줄 실행 | |
| 매매 호흡 | 일봉 기반 스윙, 하루 1회 실행 | 미국장 개장 시각(KST 22:30/23:30) |
| 시장 순서 | 미국 먼저 → 한국 확장 | |
| 유니버스 | 미국 대형주(S&P100급) 메인 + 고변동 종목 소수 혼합 | 고변동 슬리브는 별도 비중 상한으로 격리 |
| 전략 | 다중 팩터 스코어링 랭킹 (노션 문서 방식 흡수) | 아래 §4 |
| 감정점수 | 과거 데이터 백테스트 없음. 지금부터 매일 수집, 검증 기간 전방 검증 | 섀도 팩터로 시작 |
| 알림 | 텔레그램 봇 | |
| 검증 | 백테스트 → 모의투자 약 1개월 → 실계좌 | §8 |

## 3. 아키텍처

하루 1회 cron/launchd가 `run.py`를 실행:

```
run.py
 ├─ 1. KIS 토큰 발급/갱신
 ├─ 2. 일봉 데이터 업데이트 (유니버스 → SQLite 캐시)
 ├─ 3. 감정점수 수집·기록 (섀도 팩터, §4.3)
 ├─ 4. strategy.signal(data) → {심볼: 목표비중}     ← 순수 함수
 ├─ 5. 목표비중 vs 현재잔고 diff → 주문 리스트
 ├─ 6. 리스크 게이트 검사 (§5)
 ├─ 7. 주문 집행 (dry-run 모드면 기록만)
 ├─ 8. SQLite에 주문/체결/포트폴리오 스냅샷 기록
 └─ 9. 텔레그램 결과 요약 전송
```

### 모듈 구성

| 모듈 | 역할 |
|---|---|
| `broker/` | KIS REST 클라이언트 (인증, 시세, 잔고, 주문). 모의↔실전은 base URL+앱키 교체. 기존 오픈소스 래퍼(python-kis 등) 우선 검토, 안 맞으면 직접 REST 호출 |
| `data/` | 일봉 수집·SQLite 캐시. 백테스트용 장기 과거 데이터는 yfinance 보조. 감정점수 일일 수집·저장 |
| `strategy/` | 순수 함수 `signal(data) -> {symbol: weight}`. 팩터 스코어링 (§4) |
| `backtest/` | pandas 벡터 백테스터 (프레임워크 없음). 수수료·슬리피지 반영. 라이브와 동일한 strategy 함수 호출 |
| `engine/` | 목표비중→주문 변환, 집행, 기록, 멱등성 |
| `risk/` | 주문 전 게이트 (§5) |
| `notify/` | 텔레그램 전송 |

## 4. 전략 v1: 다중 팩터 스코어링 랭킹

참고 문서(노션 "매수 추천 종목 선정 기준")의 구조를 채택하되, 문서에 없던 **매도 룰**과
**검증 불가 팩터의 격리**를 추가한다.

### 4.1 매수 (진입)

매 실행일:

1. 유니버스 각 종목에 팩터 점수 계산: `score = Σ (가중치 × 팩터)`
   - **v1 팩터 (전부 가격 데이터만으로 계산 → 백테스트 가능):**
     - 60거래일 모멘텀 (수익률)
     - 골든크로스 여부 (SMA50 > SMA200)
     - RSI(14) < 50
     - MACD 매수신호
   - **섀도 팩터 (가중치 0, 기록만):** 뉴스 감정점수 (§4.3)
   - **확장 슬롯 (v1 제외):** 외부 ML 예측값 — 전방 검증으로 유효성이 입증되기 전에는 가중치를 주지 않는다
2. 점수 상위 N개 **그리고 종가 > SMA200** 인 종목을 균등 비중으로 보유
3. 조건 통과 종목이 0개면 **전량 현금** (하락장 자동 회피)

### 4.2 매도 (청산) — 셋 중 하나라도 해당하면 다음 실행 때 전량 매도

1. 점수 순위권(N위) 이탈
2. 종가가 SMA200 아래로 이탈
3. 매수가 대비 -10% 손절 (리스크 게이트에서 집행)

### 4.3 감정점수 운영 방식

- 과거 데이터를 구하지 않는다. **지금부터 매일 유니버스 전 종목의 감정점수를 수집·저장**한다.
- 검증 기간 동안 가중치 0의 **섀도 팩터**로 기록만 하고, "감정점수를 반영했다면 종목 선정이 어떻게 달라졌고 성과가 어땠는지"를 일일 비교 리포트로 누적한다.
- 약 1개월 전방 검증 후 데이터가 유효하다고 판단되면 가중치를 부여한다.
- 감정 데이터 소스는 구현 단계에서 선정 (후보: Alpha Vantage News Sentiment 등 무료 티어).

### 4.4 유니버스

- **메인 슬리브:** S&P100급 미국 대형주 (설정 파일의 종목 리스트)
- **고변동 슬리브:** 고변동성 종목 소수. **총 비중 상한 10%, 종목당 5%** 로 격리
  - 근거: 하루 1회 실행 구조에서는 밤새 갭 리스크를 장중 대응으로 막을 수 없으므로, 비중 상한이 유일한 실효 방어선이다
- 유니버스는 코드가 아닌 설정 파일이므로 교체·확장에 코드 변경 불필요

### 4.5 파라미터 (전부 설정 파일, 백테스트로 확정)

초기값 제안: N=3~5, 모멘텀 60일, SMA 50/200, RSI 14, 손절 -10%, 팩터 가중치는
백테스트 그리드로 결정. **이 숫자들은 제안값이며 Phase 1 백테스트가 확정한다.**

## 5. 리스크 관리 (실계좌 필수 요건, 축소 금지)

- 종목당 최대 비중 / 최소 주문 금액
- 고변동 슬리브 비중 상한 (§4.4)
- 일일 손실 한도 초과 시 신규 주문 전면 중지 + 텔레그램 경보
- **kill switch:** 지정 파일이 존재하면 무조건 아무것도 하지 않음
- **dry-run 모드:** 주문 API만 스킵하고 전 과정 기록. 실계좌 전환 전 관문
- fail-safe: 잔고/체결 확인 실패 등 상태 불확실 시 거래하지 않고 경보

## 6. 데이터 (SQLite 단일 파일)

- `prices` — 일봉 OHLCV
- `sentiment` — 일일 감정점수 (종목, 날짜, 점수, 소스)
- `orders` — 주문 기록 (실행 ID, 멱등성 키 포함)
- `fills` — 체결 기록
- `snapshots` — 일일 포트폴리오 스냅샷 (전방 검증·성과 분석용)
- `runs` — 실행 로그 (시작/종료, 결과, 오류)

## 7. 에러 처리

- **주문 멱등성:** 실행 ID 기반. 재실행 시 이미 낸 주문 중복 발행 금지
- API 일시 오류 재시도, 지속 실패 시 텔레그램 경보 후 안전 종료
- 모든 실행의 전체 로그를 파일+DB에 보존 (감사 기록)

## 8. 검증 로드맵 (개발 순서와 동일)

1. **Phase 1 — 백테스터 + 전략:** 과거 10년+ 일봉으로 v1 팩터·파라미터 확정
2. **Phase 2 — KIS 모의투자 자동매매 (약 1개월):**
   - 모의 서버로 전체 주문 흐름 검증
   - 감정점수 매일 수집 + 섀도 팩터 전방 검증
3. **Phase 3 — 실계좌:** dry-run 병행 → 소액 → 본 운용
4. **이후 — 한국시장 확장:** broker에 국내주식 주문 API 추가 + 유니버스 확장

테스트: 전략 함수 단위 테스트, 백테스터 기지값 검증, 모의투자 통합 테스트.

## 9. 사전 준비물 (사용자 직접)

- 한국투자증권 계좌 개설
- KIS Developers 앱키 발급 (실전 + 모의투자)
- 텔레그램 봇 토큰 발급

## 10. 명시적으로 하지 않는 것 (YAGNI)

- 웹 대시보드 (텔레그램으로 충분, 필요해지면 2차)
- 실시간 웹소켓/장중 대응 (일봉 전략에 불필요)
- 백테스트 프레임워크 도입 (라이브와 코드 경로가 갈라지는 위험 > 편의)
- ML 예측 모델 자체 개발 (확장 슬롯만 유지)

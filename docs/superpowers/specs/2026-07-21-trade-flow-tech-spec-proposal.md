## 0. 정의 (Definitions)

### 0.1 도메인 용어

| 용어 | 정의 |
|------|------|
| 거래일 D | 주문을 제출하는 미국 정규장 거래일이다. |
| 신호 기준일 | D 직전의 완료된 미국 정규장 거래일이다. |
| 신호 데이터 | 신호 기준일까지 확정된 일봉 데이터만 포함한 입력이다. |
| 목표 포트폴리오 | 전략과 리스크 정책을 모두 적용한 종목별 목표 비중과 현금 비중이다. |
| 주문 계획 | 목표 포트폴리오와 브로커 잔고·미체결 주문의 차이로 생성한 주문 집합이다. |
| 실행 ID | 한 번의 프로세스 실행을 추적하는 UUID이다. 멱등성 판단에는 사용하지 않는다. |
| 주문 의도 ID | `account_id/trading_date/strategy_version/symbol/side/rebalance_seq`로 생성한 결정적 식별자이다. |
| 신규 매수 | 현재 보유 수량을 증가시키는 주문이다. |
| 축소 매도 | 목표 비중 또는 리스크 제한을 맞추기 위해 보유 수량을 줄이는 주문이다. |
| 위험 레짐 | VIX 또는 WTI 조건이 발동하여 포트폴리오 노출 한도를 축소한 상태이다. |
| 데이터 신선도 | 각 데이터가 신호 기준일까지 완결되었고 공급자 타임스탬프가 허용 범위 안인 상태이다. |
| 치명 오류 | 중복 주문, 반대 방향 주문, 허용 한도 초과 주문, 계좌 불일치 주문이다. |

### 0.3 불변식 (Invariants)

| Invariant | 설명 |
|-----------|------|
| 전략은 순수 함수이다 | 동일한 입력과 설정은 동일한 목표 비중을 반환하며 네트워크·DB·현재 시각을 읽지 않는다. |
| 미래 데이터를 사용하지 않는다 | 신호 계산에는 신호 기준일 이후의 가격·감정·체결 정보를 사용하지 않는다. |
| 불확실한 상태에서는 거래하지 않는다 | 잔고, 미체결 주문, 데이터 신선도, 계좌 식별 중 하나라도 확인할 수 없으면 주문을 제출하지 않는다. |
| 매도가 매수보다 먼저다 | 축소 매도를 체결·정산한 후 실제 주문 가능 현금으로 신규 매수를 계산한다. |
| 주문 의도는 멱등하다 | 같은 주문 의도 ID는 프로세스 재실행과 장애 복구 중 한 번만 브로커에 제출한다. |
| 브로커 상태가 체결의 원장이다 | 로컬 DB와 브로커가 다르면 신규 주문을 중단하고 브로커 조회 결과로 조정 기록을 남긴다. |
| 실계좌는 명시적으로 활성화한다 | 실계좌 ID allowlist, production 환경값, kill switch 해제 조건이 모두 충족되어야 한다. |
| 공매도와 레버리지를 사용하지 않는다 | 보유 수량을 초과하는 매도, 신용, 마진, 옵션 주문을 생성하지 않는다. |
| 정수 수량만 주문한다 | 소수점 주식은 사용하지 않으며 목표 비중의 잔여 금액은 현금으로 유지한다. |
| 일봉 손절은 장중 손실을 보장하지 않는다 | 손절은 신호 기준일 종가로 판정하고 D의 지정가 주문으로만 실행한다. |
| 뉴스 해석은 주문에 관여하지 않는다 | 감정점수와 뉴스 요약은 별도 승인 전까지 목표 포트폴리오를 변경하지 않는다. |
| 모든 시각은 명시적 시간대를 가진다 | 시장 이벤트는 America/New_York, 저장 시각은 UTC, 사용자 표시는 Asia/Seoul을 사용한다. |

## 1. 개요 (Overview)

### 1.1 배경 및 목적

한국투자증권 Open API를 이용하여 미국주식 일봉 스윙 전략을 백테스트, 모의투자, 실계좌에서 동일한 전략 계약으로 실행한다. 프로그램은 거래일마다 한 번 실행하고 주문·체결·포트폴리오 상태를 재현 가능한 감사 기록으로 남긴다.

### 1.2 목표

- 백테스트와 라이브 실행기가 동일한 전략 함수를 호출한다.
- 주문 전에 데이터, 계좌, 리스크, 중복 여부를 검증한다.
- 장애 후 재실행해도 중복 주문이 발생하지 않는다.
- 전략 성과와 운영 안정성을 별도 관문으로 검증한다.
- 미국시장 검증이 끝난 뒤 브로커 경계를 유지한 채 한국시장으로 확장한다.

### 1.3 비목표

- 장중 실시간 손절과 웹소켓 시세 처리는 포함하지 않는다.
- 공매도, 레버리지, 옵션, 소수점 매매는 포함하지 않는다.
- 자동 환전과 통합증거금 자동 사용은 포함하지 않는다.
- 뉴스 또는 LLM의 해석으로 주문을 생성하지 않는다.
- 웹 대시보드와 다중 사용자 권한 시스템은 포함하지 않는다.
- 외부 ML 모델 개발은 포함하지 않는다.
- 수익 또는 원금 보장을 목표로 하지 않는다.

|------|------|------|

| In Scope | 전략 계산 | 가격 기반 팩터를 계산해 목표 포트폴리오를 만든다. |
| In Scope | 백테스트 | 동일 전략 계약으로 비용·슬리피지·다음 날 체결을 시뮬레이션한다. |
| In Scope | 브로커 연동 | KIS 인증, 계좌, 잔고, 주문, 체결 조회를 캡슐화한다. |
| In Scope | 주문 실행 | 매도 우선, 지정가, 미체결 취소, 멱등성, reconciliation을 처리한다. |

## 3. 기능 명세 (Functional Specification)

### 3.1 실행 오케스트레이션

**전제 조건**

- 실행 환경이 `backtest`, `paper`, `dry-run`, `production` 중 하나로 명시되어 있다.
- 대상 계좌가 해당 환경의 allowlist와 일치한다.
- 미국시장 캘린더가 거래일 D와 신호 기준일을 반환한다.

**정상 흐름**

| 단계 | 동작 | 결과 |
|------|------|------|
| 1 | 프로세스 락을 획득하고 실행 ID를 생성한다. | 중복 프로세스가 차단된다. |
| 2 | kill switch, 환경, 계좌 ID를 검증한다. | 주문 가능 여부가 확정된다. |
| 3 | 브로커 잔고와 미체결 주문을 조회해 로컬 상태와 조정한다. | 시작 상태가 확정된다. |
| 4 | 시장 캘린더와 실행 허용 지연을 검사한다. | 거래일과 실행 가능 시간이 확정된다. |
| 5 | 신호 데이터와 레짐 데이터를 수집하고 신선도를 검증한다. | 전략 입력 스냅샷이 생성된다. |
| 6 | 전략이 원시 목표 비중을 계산한다. | 가격 기반 후보와 비중이 생성된다. |
| 7 | 리스크 정책이 손절과 노출 한도를 반영한다. | 목표 포트폴리오가 확정된다. |
| 8 | 현재 상태와 목표 포트폴리오의 차이로 주문 계획을 만든다. | 매도·매수 주문 의도가 생성된다. |
| 9 | 매도를 먼저 실행하고 브로커 상태를 다시 조회한다. | 실제 주문 가능 현금이 확정된다. |
| 10 | 현금 범위에서 매수 수량을 다시 계산하고 실행한다. | 목표 포트폴리오에 근접한다. |
| 11 | 미체결 주문을 취소하고 최종 체결·잔고를 조정한다. | 종료 스냅샷이 확정된다. |
| 12 | DB와 파일 로그를 기록하고 알림을 전송한다. | 실행 결과가 감사 가능하게 보존된다. |

**예외 흐름**

| 조건 | 처리 방식 |
|------|----------|
| 휴장일이다 | 주문 없이 성공 종료하고 실행 사유를 기록한다. |
| 실행 허용 지연을 초과했다 | 주문 없이 실패 종료하고 경보를 전송한다. |
| 락이 이미 존재한다 | 기존 프로세스 생존 여부를 확인하고, 생존 중이면 중복 실행으로 종료한다. |
| 이전 프로세스가 남긴 stale lock이다 | 기존 실행 상태를 조정한 뒤 새 락을 획득한다. |
| 데이터가 누락되거나 오래되었다 | 신규 주문과 축소 매도를 모두 중단하고 경보를 전송한다. |
| 잔고 또는 미체결 주문 조회가 실패한다 | 주문 없이 실패 종료한다. |
| 텔레그램 전송이 실패한다 | 실행 결과는 유지하고 로그에 경보 실패를 기록해 scheduler 감시 대상으로 노출한다. |
| 처리되지 않은 예외가 발생한다 | 비정상 종료 코드를 반환하고 가능한 범위에서 실행 실패를 기록한다. |

### 3.2 데이터 수집과 검증

**입력**

- 설정 파일의 유니버스와 공급자 심볼 매핑
- 미국시장 캘린더
- 가격·레짐·감정 데이터 공급자

**처리 규칙**

- OHLCV는 신호 기준일까지의 행만 전략 입력에 포함한다.
- 백테스트 신호에는 split-adjusted OHLC를 사용하고 배당은 포트폴리오 수익에 현금으로 반영한다.
- 라이브 데이터에는 원본 가격과 공급자 수정 여부를 함께 저장한다.
- 심볼 변경, 분할, 상장폐지는 `instruments` 이력으로 관리한다.
- 데이터 공급자 심볼과 KIS 주문 심볼을 별도 필드로 관리한다.
- 가격 역전, 음수 거래량, 중복 키, 캘린더 불일치를 유효성 오류로 처리한다.
- 레짐 입력이 유효하지 않으면 안전 상태로 전환하고 신규 매수를 금지한다.

**출력**

- `MarketDataSnapshot(as_of, prices, regime_inputs, quality_report)`
- 품질 보고서는 누락, 중복, 최신 거래일, 공급자 타임스탬프를 포함한다.

### 3.3 가격 기반 전략

**팩터 정의**

| 팩터 | 값의 정의 | 정규화 |
|------|-------------|--------|
| 모멘텀 | 신호 기준일 종가 ÷ 60거래일 전 종가 - 1 | 적격 종목 내 percentile rank 0~1 |
| 추세 | SMA50이 SMA200보다 크면 1, 아니면 0 | 없음 |
| RSI | RSI14가 50 미만이면 `(50-RSI14)/50`, 아니면 0 | 0~1 범위 제한 |
| MACD | EMA12-EMA26이 signal EMA9보다 크면 1, 아니면 0 | 없음 |

**적격 조건**

- 신호 기준일 종가가 SMA200보다 크다.
- 일봉 최소 길이와 데이터 신선도 조건을 만족한다.
- 거래정지, 상장폐지 진행, 주문 불가 심볼이 아니다.
- 가격이 0보다 크고 정수 1주 이상 주문할 수 있다.

**선정과 비중**

- 총점은 Section 0.2의 팩터 가중치와 팩터 값의 가중합이다.
- 동점이면 모멘텀, Section 0.2의 유동성 동점 판정 기간 평균 거래대금, 심볼 오름차순으로 결정한다.
- 메인 슬리브는 적격 종목 중 상위 N개를 같은 목표 비중으로 배분한다.
- 고변동 슬리브는 별도 설정 목록에서 상위 종목을 선정하고 슬리브 한도를 적용한다.
- 적격 종목 수가 N보다 적으면 남는 비중은 현금으로 유지한다.
- 전략 출력은 음수 비중을 포함하지 않으며 전체 합이 1을 넘지 않는다.

**출력 계약**

```python
@dataclass(frozen=True)
class StrategyResult:
    as_of: date
    target_weights: Mapping[str, Decimal]
    cash_weight: Decimal
    scores: Mapping[str, FactorScore]
    exclusions: Mapping[str, str]

def signal(snapshot: MarketDataSnapshot, config: StrategyConfig) -> StrategyResult: ...
```

### 3.4 위험 정책

- 손절 대상은 전략 순위와 무관하게 목표 비중을 0으로 만든다.
- 위험 레짐에서는 총 주식 목표 비중이 위험 레짐 현금 하한을 넘지 않도록 모든 슬리브 비중을 비례 축소한다.
- 일일 손실 한도가 발동하면 신규 매수를 금지하고 축소 매도와 손절 매도만 허용한다.
- 종목별·슬리브별 상한을 초과한 목표 비중은 상한까지 축소하고 차액은 현금으로 유지한다.
- 달러 주문 가능 금액이 부족하면 매수 목표를 비례 축소하며 매도 예상대금은 실제 체결 전까지 사용하지 않는다.
- 현재가, 잔고, 평균 체결단가 중 하나라도 없으면 해당 종목의 주문을 생성하지 않는다.

**출력 계약**

```python
def apply_risk_policy(
    strategy_result: StrategyResult,
    account: AccountSnapshot,
    regime: RegimeState,
    config: RiskConfig,
) -> RiskAdjustedTarget: ...
```

### 3.5 주문 계획과 집행

**주문 수량**

- 목표 금액은 확정 NAV와 목표 비중의 곱이다.
- 목표 수량은 목표 금액을 기준 가격으로 나눈 값의 내림 정수이다.
- 현재 수량과 목표 수량의 차이가 주문 수량이다.
- 수수료와 Section 0.2의 현금 하한을 제외한 주문 가능 현금 안에서만 매수한다.
- 최소 주문 금액 미만 차이는 주문하지 않고 drift로 기록한다.

**지정가와 체결**

- 기준 가격은 주문 직전 조회한 유효한 최우선 호가이며 호가가 없으면 최근 체결가를 사용한다.
- 지정가는 Section 0.2의 허용 폭을 적용하고 거래소 호가 단위로 보수적으로 반올림한다.
- 주문 제출 전에 주문 의도 ID로 로컬 DB와 브로커 주문을 함께 조회한다.
- 주문 유효시간 동안 체결 상태를 조회하고 남은 수량은 취소한다.
- 부분 체결 수량만 잔고와 현금에 반영한다.
- 취소 확인이 실패하면 후속 매수를 금지하고 수동 확인 경보를 전송한다.
- 다음 실행은 남은 목표 차이를 새 시장 상태로 다시 계산하며 이전 주문을 이월하지 않는다.

**재실행과 복구**

- 시작 시 전일 이후의 미체결·체결 주문을 브로커에서 조회한다.
- 주문 의도 ID가 이미 제출된 경우 새 주문을 만들지 않고 기존 주문 상태를 추적한다.
- 브로커가 사용자 정의 주문 ID를 지원하지 않으면 주문 시각, 심볼, 방향, 수량, 가격을 조합한 보조 대조키를 저장한다.
- 주문 응답을 받지 못한 네트워크 타임아웃은 즉시 재주문하지 않고 브로커 주문 조회로 존재 여부를 확인한다.

### 3.6 감정점수 관찰

- Section 0.2의 감정점수 수집 후보 상한까지만 수집 대상으로 삼는다.
- 기사별 티커 관련도와 감정점수의 가중 평균, 기사 수, 결측 사유를 저장한다.
- 기사가 없는 날은 0점이 아니라 `missing`으로 저장한다.
- 감정점수는 목표 비중과 주문 계획에 전달하지 않는다.
- 관찰 기간 종료 후 사전 정의된 평가 보고서를 생성하고 별도 승인 문서에서 반영 여부를 결정한다.
- 평가 보고서는 커버리지, Section 0.2의 감정점수 평가 수익 구간과의 rank IC, 포트폴리오 교체율, 비용 차감 성과를 포함한다.

### 3.7 알림과 운영 상태

**일일 결과 알림**

- 실행 ID와 환경
- 신호 기준일과 데이터 품질
- 레짐 상태와 입력값
- 목표·실제 포지션 차이
- 제출·체결·취소·실패 주문
- 현금, NAV, 일일 손익
- 감정점수 관찰 요약
- 수동 확인 필요 여부

**운영 감시**

- scheduler는 프로세스 종료 코드와 최신 성공 실행 시각을 감시한다.
- 거래일의 실행 허용 시간이 끝날 때 성공 또는 명시적 휴장 기록이 없으면 외부 dead-man 경보를 발생시킨다.
- kill switch가 활성화된 실행도 주문 없이 상태와 경보를 기록한다.

## 6. 외부 인터페이스 (External Interfaces)

### 6.1 브로커 계약

```python
class Broker(Protocol):
    def account_snapshot(self) -> AccountSnapshot: ...
    def quote(self, symbol: str) -> Quote: ...

    def submit(self, request: OrderRequest) -> BrokerOrder: ...
```

- paper와 production 구현은 동일 계약을 사용하고 base URL, 앱키, 계좌 allowlist를 분리한다.
- 인증 헤더와 원문 응답의 민감 필드는 저장 전에 마스킹한다.
- HTTP 성공 여부와 업무 오류 코드를 분리해 처리한다.
- 조회 API만 자동 재시도하며 주문 API 타임아웃은 Section 3.5의 대조 절차를 따른다.

### 6.2 시장 데이터 계약

```python
class MarketDataProvider(Protocol):
    def daily_bars(self, symbol: str, start: date, end: date) -> Sequence[DailyBar]: ...
    def regime_inputs(self, start: date, end: date) -> Sequence[RegimeInput]: ...
```

- 공급자별 심볼, 수정주가, 캘린더 의미를 adapter가 canonical 모델로 변환한다.
- 백테스트 데이터와 라이브 데이터의 차이를 일별 검증 리포트로 기록한다.

### 6.3 감정 데이터 계약

```python
class SentimentProvider(Protocol):
    def daily_sentiment(self, symbol: str, session_date: date) -> SentimentObservation: ...
```

- rate limit을 넘지 않도록 일일 요청 예산을 설정한다.

### 6.4 알림 계약

```python
class Notifier(Protocol):
```

- 알림 실패는 DB와 프로세스 종료 상태로 외부 감시에 전달한다.

## 7. 검증 및 출시 기준 (Validation & Rollout)

### 7.1 Phase 1 — 백테스트와 전략 승인

**데이터와 방법**

- Section 0.2의 백테스트 최소 기간을 포함하고 holdout은 파라미터 선택에 사용하지 않는다.
- 나머지 기간은 Section 0.2의 walk-forward 구간으로 평가한다.
- point-in-time 유니버스를 기본 결과로 사용한다.
- point-in-time 데이터를 구하지 못하면 현재 유니버스 결과를 참고용으로만 표시하고 실계좌 승인 근거로 사용하지 않는다.
- SPY buy-and-hold와 100% 현금을 비교 기준으로 사용한다.
- 거래비용은 Section 0.2의 거래비용 민감도 결과를 모두 제시한다.
- 전체 기간과 주요 충격 구간의 CAGR, MDD, Sharpe, Calmar, 회전율, 승률, 회복기간을 보고한다.

**승인 기준**

- 미래 참조, 중복 봉, 잘못된 수정주가 사용이 0건이다.
- holdout에서 비용 차감 전략 MDD가 SPY보다 작다.
- Section 0.2의 가장 보수적인 비용 시나리오에서 전략의 장기 기대수익이 현금보다 높다.
- 인접 파라미터 조합에서도 성과 방향이 유지되어 단일 최적점 의존이 없다.
- 전략 결과가 고정 fixture와 일치하고 동일 입력에서 결정적이다.
- 승인된 파라미터와 데이터 해시는 변경 불가능한 실행 manifest로 보존한다.

### 7.2 Phase 2 — dry-run과 모의투자 승인

- Section 0.2의 모의 전 dry-run 기간에는 production 자격증명을 사용하지 않는다.
- 이후 Section 0.2의 모의투자 최소 기간 동안 연속 모의투자를 수행한다.
- 치명 오류가 0건이다.
- 제출한 모든 주문 의도가 브로커 주문 또는 명시적 미제출 사유와 연결된다.
- 강제 종료와 네트워크 타임아웃 장애 주입 후 중복 주문이 0건이다.
- 일별 목표 수량과 최종 수량 차이가 정수 반올림, 최소 주문 금액, 미체결로 설명된다.
- kill switch, stale data, 잔고 조회 실패, 일일 손실 한도, 알림 실패를 각각 한 번 이상 검증한다.
- 거래일 실행 누락이 모두 dead-man 경보로 감지된다.

### 7.3 Phase 3 — 실계좌 단계적 출시

- Section 0.2의 실계좌 dry-run 기간에는 production 조회와 dry-run만 수행한다.
- 다음 Section 0.2의 실계좌 소액 검증 기간은 별도 승인된 자본 상한으로 운용한다.
- 매 거래일 사용자가 일일 리포트와 조정 결과를 확인한다.
- 치명 오류 또는 설명할 수 없는 잔고 차이가 한 번이라도 발생하면 신규 매수를 중단하고 Phase 2로 되돌린다.
- 자본 상한 확대는 별도 승인 기록과 설정 변경으로만 수행한다.

### 7.4 테스트 범위

- 팩터 경계값, 결측값, 동점, 분할 데이터를 포함한 전략 단위 테스트
- 미래 참조와 다음 날 체결을 검증하는 백테스터 golden fixture 테스트
- 수량 내림, 현금 부족, 비중 상한, 손절의 속성 기반 테스트
- 브로커 오류 코드, rate limit, 타임아웃의 contract 테스트
- 주문 응답 유실, 부분 체결, 취소 실패, 강제 종료의 장애 주입 테스트
- DB migration, 무결성 검사, 백업 복구 테스트
- paper와 production 설정 격리 테스트

## 0. 정의 (Definitions)

### 0.1 도메인 용어

| 용어 | 정의 |
|------|------|
| 거래일 D | 주문을 제출하는 미국 정규장 거래일이다. |
| 신호 기준일 | D 직전의 완료된 미국 정규장 거래일이다. |
| 신호 데이터 | 신호 기준일까지 확정된 일봉 데이터만 포함한 전략 입력이다. |
| 유니버스 등급 A | 공식 또는 상용 데이터로 각 거래일의 실제 구성 종목을 재현한 point-in-time 유니버스이다. |
| 유니버스 등급 B | 공개된 편입·제외 이력으로 구성 종목을 근사 복원하고 출처와 누락을 기록한 유니버스이다. |
| 유니버스 등급 C | 현재 구성 종목을 전체 과거 기간에 적용한 생존 편향 유니버스이다. |
| 원시 목표 비중 | 가격 기반 전략만 계산한 종목별 목표 비중이다. |
| 최종 목표 비중 | 리스크 정책을 적용한 종목별 목표 비중과 현금 비중이다. |
| 실행 ID | 한 번의 프로세스 실행을 추적하는 UUID이다. |
| 주문 의도 ID | `account_id/trading_date/strategy_version/symbol/side/rebalance_seq`로 생성한 결정적 식별자이다. |
| 신규 매수 중단 정책 | 위험 레짐 동안 모든 매수 주문을 금지하고 전략 청산과 손절 매도만 허용하는 정책이다. |
| 노출 축소 정책 | 위험 레짐 동안 주식 목표 비중을 정해진 상한까지 비례 축소하는 실험 정책이다. |
| 데이터 신선도 | 신호 데이터가 신호 기준일까지 완결되고 공급자 타임스탬프와 시장 캘린더가 일치하는 상태이다. |
| 치명 오류 | 중복 주문, 반대 방향 주문, 보유 수량 초과 매도, 리스크 한도 초과 주문, 계좌 불일치 주문이다. |

### 0.3 불변식 (Invariants)

| Invariant | 설명 |
|-----------|------|
| 전략은 순수 함수이다 | 동일한 입력과 설정은 의미상 동일한 점수, 순위, 목표 비중을 반환한다. |
| 미래 데이터를 사용하지 않는다 | 신호 계산에는 신호 기준일 이후의 가격·감정·체결 정보를 사용하지 않는다. |
| 불확실하면 거래하지 않는다 | 잔고, 미체결 주문, 데이터 신선도, 계좌 중 하나라도 확인할 수 없으면 주문하지 않는다. |
| 주문 상태는 브로커가 원장이다 | 로컬 DB와 브로커가 다르면 신규 주문을 중단하고 브로커 상태로 조정한다. |
| 매도를 먼저 처리한다 | 매도 체결과 취소를 확인한 뒤 실제 주문 가능 현금으로 매수 수량을 계산한다. |
| 주문 의도는 멱등하다 | 같은 주문 의도 ID는 재실행과 장애 복구 과정에서 한 번만 제출한다. |
| 실계좌는 명시적으로 활성화한다 | production 환경, 계좌 allowlist, kill switch 해제를 모두 확인한다. |
| 공매도와 레버리지를 사용하지 않는다 | 보유 수량 초과 매도, 신용, 마진, 옵션 주문을 생성하지 않는다. |
| 정수 수량만 주문한다 | 소수점 주식을 사용하지 않고 잔여 금액은 현금으로 유지한다. |
| 레짐 기본 동작은 신규 매수 중단이다 | 노출 축소 정책은 Phase 1 승인 전에는 paper와 production 주문에 적용하지 않는다. |
| 일봉 손절은 장중 손실을 보장하지 않는다 | 손절은 신호 기준일 종가로 판정하고 D의 주문으로 실행한다. |
| 감정점수는 주문에 관여하지 않는다 | 별도 승인 전에는 감정점수를 전략 입력과 최종 목표 비중에 전달하지 않는다. |
| 시간대 없는 시각을 사용하지 않는다 | 시장 이벤트는 America/New_York, 저장은 UTC, 사용자 표시는 Asia/Seoul을 사용한다. |
| 생존 편향과 거래비용을 분리 평가한다 | 높은 거래비용 가정으로 유니버스 생존 편향을 대체하지 않는다. |

## 1. 개요 (Overview)

### 1.1 배경 및 목적

한국투자증권 Open API 기반의 미국주식 일봉 스윙 자동매매 시스템을 구축한다. 백테스트와 라이브 실행기는 동일한 전략 계약을 사용하며, 프로그램은 미국 거래일마다 한 번 실행하고 종료한다.

### 1.2 목표

- 가격 기반 다중 팩터 전략을 미래 참조 없이 검증한다.
- 목표 비중부터 주문·체결·최종 잔고까지 추적 가능한 기록을 남긴다.
- 네트워크 장애와 프로세스 재실행에서 중복 주문을 방지한다.
- dry-run과 모의투자를 거친 뒤 승인된 소액으로 실계좌를 검증한다.
- 감정점수와 위험 레짐의 효과를 주문 정책과 분리해 평가한다.

### 1.3 비목표

- 장중 실시간 손절과 웹소켓 시세 처리는 포함하지 않는다.
- 공매도, 레버리지, 옵션, 소수점 매매는 포함하지 않는다.
- 자동 환전과 통합증거금 자동 사용은 포함하지 않는다.
- 뉴스 또는 LLM 해석으로 주문을 생성하지 않는다.
- 웹 대시보드와 다중 사용자 권한 시스템은 포함하지 않는다.
- 외부 ML 모델 개발과 한국주식 거래는 포함하지 않는다.

## 3. 기능 명세 (Functional Specification)

### 3.1 일일 실행

**전제 조건**

- 환경은 `backtest`, `paper`, `dry-run`, `production` 중 하나이다.
- 대상 계좌가 환경별 allowlist와 일치한다.
- 거래일 D와 신호 기준일을 미국시장 캘린더로 확인할 수 있다.

**정상 흐름**

| 단계 | 동작 | 결과 |
|------|------|------|
| 1 | 프로세스 락을 획득하고 실행 ID를 생성한다. | 중복 프로세스를 차단한다. |
| 2 | 환경, 계좌, kill switch를 검사한다. | 주문 가능 여부를 확정한다. |
| 3 | 브로커의 잔고와 미체결 주문을 로컬 상태와 조정한다. | 시작 계좌 상태를 확정한다. |
| 4 | 거래일, 실행 시각, 데이터 신선도를 검사한다. | 전략 입력 가능 여부를 확정한다. |
| 5 | 전략이 원시 목표 비중을 계산한다. | 후보, 점수, 목표 비중을 생성한다. |
| 6 | 리스크 정책이 최종 목표 비중을 계산한다. | 손절과 주문 제한을 반영한다. |
| 7 | 현재 잔고와 최종 목표 비중의 차이로 주문 의도를 만든다. | 매도와 매수 계획을 생성한다. |
| 8 | 매도를 제출하고 체결·취소 상태를 조정한다. | 실제 주문 가능 현금을 확정한다. |
| 9 | 현금 범위에서 매수 수량을 다시 계산하고 제출한다. | 허용 범위에서 목표 비중에 접근한다. |
| 10 | 미체결 주문을 취소하고 최종 잔고를 조정한다. | 실행 결과를 확정한다. |
| 11 | DB와 로그를 기록하고 텔레그램 알림을 전송한다. | 감사 기록을 완료한다. |

**예외 흐름**

| 조건 | 처리 |
|------|------|
| 휴장일이다 | 주문 없이 성공 종료하고 휴장 사유를 기록한다. |
| 허용 시각을 넘겼다 | 주문 없이 실패 종료하고 경보를 기록한다. |
| 살아 있는 프로세스의 락이 있다 | 중복 실행으로 종료한다. |
| 종료된 프로세스의 락이 남았다 | 이전 실행 상태를 조정한 후 락을 교체한다. |
| 데이터가 누락되거나 오래되었다 | 모든 주문을 차단하고 경보를 기록한다. |
| 잔고 또는 미체결 주문 조회에 실패했다 | 모든 주문을 차단하고 실패 종료한다. |
| 텔레그램 전송에 실패했다 | 거래 결과를 유지하고 실행 상태에 알림 실패를 기록한다. |
| 처리되지 않은 예외가 발생했다 | 비정상 종료 코드를 반환하고 가능한 실행 상태를 기록한다. |

### 3.2 데이터 수집과 품질

- OHLCV는 신호 기준일까지 완료된 행만 전략에 전달한다.
- 백테스트 신호에는 split-adjusted 가격을 사용하고 배당은 포트폴리오 현금흐름에 반영한다.
- 원본 심볼, 데이터 공급자 심볼, KIS 주문 심볼은 설정 파일에서 매핑한다.
- 티커 변경, 분할, 상장폐지는 발견 시 설정 파일의 유효기간 매핑으로 추가한다.
- 가격 역전, 음수 거래량, 중복 행, 시장 캘린더 불일치를 데이터 오류로 처리한다.
- 레짐 데이터가 유효하지 않으면 신규 매수를 차단한다.
- 수집 결과는 최신 거래일, 누락, 중복, 공급자 시각을 포함한 품질 보고서를 반환한다.

### 3.3 가격 기반 전략

**팩터 계약**

| 팩터 | 계산 | 점수 범위 |
|------|------|-----------|
| 모멘텀 | 신호 기준일 종가 ÷ Section 0.2의 모멘텀 기간 전 종가 - 1의 유니버스 내 percentile rank | 0~1 |
| 추세 | Section 0.2의 단기 SMA > 장기 SMA이면 1, 아니면 0 | 0 또는 1 |
| RSI | Section 0.2의 RSI 신호를 만족하면 1, 아니면 0 | 0 또는 1 |
| MACD | Section 0.2의 MACD 신호를 만족하면 1, 아니면 0 | 0 또는 1 |

**적격 조건과 비중**

- 신호 기준일 종가가 SMA200보다 크다.
- 최소 가격 데이터와 데이터 신선도를 충족한다.
- 거래정지, 상장폐지 진행, 주문 불가 심볼이 아니다.
- 총점은 Section 0.2의 팩터 가중치와 점수의 가중합이다.
- 동점은 모멘텀, Section 0.2 기간의 평균 거래대금, 심볼 오름차순으로 결정한다.
- 메인 슬리브는 상위 N개에 Section 0.2의 총 목표 비중을 균등 배분한다.
- 고변동 슬리브는 별도 목록에서 선정하고 Section 0.2의 상한을 적용한다.
- 적격 종목이 부족하거나 정수 수량을 만들 수 없으면 남은 금액을 현금으로 유지한다.

**출력 계약**

```python
@dataclass(frozen=True)
class StrategyResult:
    as_of: date
    target_weights: Mapping[str, Decimal]
    cash_weight: Decimal
    scores: Mapping[str, FactorScore]
    exclusions: Mapping[str, str]

def signal(snapshot: MarketDataSnapshot, config: StrategyConfig) -> StrategyResult: ...
```

### 3.4 위험 레짐과 리스크 정책

- 위험 레짐의 주문 기본 동작은 신규 매수 중단 정책이다.
- 위험 레짐에서도 전략 청산, 손절 매도, 비중 상한 초과분 매도는 허용한다.
- 노출 축소 정책은 백테스트 결과가 승인되기 전까지 주문에 적용하지 않는다.
- 손절 대상은 전략 순위와 무관하게 최종 목표 비중을 0으로 만든다.
- 일일 손실 한도가 발동하면 매수를 금지하고 축소·손절 매도만 허용한다.
- 종목별·슬리브별 상한 초과분은 현금으로 유지한다.
- 달러 예수금이 부족하면 매수 수량을 줄이고 경보를 기록한다.

### 3.5 주문과 장애 복구

- 목표 수량은 목표 금액을 유효한 기준 가격으로 나눈 내림 정수이다.
- 주문 현금 여유분과 예상 수수료를 제외한 주문 가능 현금 안에서만 매수한다.
- 주문 직전의 최우선 호가를 기준 가격으로 사용하고 호가가 없으면 주문을 만들지 않는다.
- 지정가는 허용 폭을 적용하고 거래소 호가 단위로 보수적으로 반올림한다.
- 주문 의도는 주문 API 호출 전에 DB에 먼저 저장한다.
- 주문 제출 전에 주문 의도 ID로 로컬 주문과 브로커 주문을 조회한다.
- 주문 API 타임아웃은 즉시 재주문하지 않고 브로커 주문 조회로 접수 여부를 확인한다.
- 주문 유효시간 동안 체결을 조회하고 남은 수량을 취소한다.
- 부분 체결 수량만 잔고와 현금에 반영한다.
- 취소 완료를 확인하지 못하면 후속 매수를 차단하고 수동 확인 상태로 종료한다.
- 다음 실행은 미체결 수량을 이월하지 않고 최신 목표 비중으로 다시 계산한다.

### 3.6 감정점수 관찰

- Section 0.2의 후보 상한 안에서만 감정점수를 수집한다.
- 기사별 티커 관련도와 감정점수의 가중 평균, 기사 수, 결측 사유를 저장한다.
- 기사 없음, 공급자 오류, 호출 한도 초과를 서로 다른 결측값으로 기록한다.
- 기사 없음은 중립 점수로 변환하지 않는다.
- 평가 보고서는 커버리지, Section 0.2의 평가 구간 rank IC, 교체율, 비용 차감 성과를 포함한다.

### 3.7 운영 알림

- 일일 알림은 환경, 실행 ID, 신호 기준일, 데이터 품질, 레짐, 주문, 체결, 취소, NAV, 현금, 수동 확인 여부를 포함한다.
- 거래일 실행 누락은 프로세스 외부 scheduler의 최신 성공 시각 검사로 감지한다.
- kill switch 실행도 주문 없이 상태와 알림을 기록한다.
- 알림 실패는 `runs`에 기록하고 프로세스 종료 상태로 외부 감시에 노출한다.

## 5. 데이터 모델 (Data Model)

금액, 가격, 수량 계산에는 이진 부동소수점을 사용하지 않는다. 초기 버전은 아래 테이블만 사용하고 심볼 이력과 알림은 설정 파일과 `runs` 필드로 관리한다.

| 테이블 | 주요 필드 | 제약조건 |
|--------|-----------|----------|
| `prices` | symbol, session_date, OHLCV, adjusted_close, source, fetched_at | `(symbol, session_date, source)` unique |
| `sentiment` | symbol, session_date, score, relevance, article_count, source, missing_reason | `(symbol, session_date, source)` unique |
| `orders` | intent_id, run_id, broker_order_id, symbol, side, requested_qty, limit_price, status, error_code | intent_id unique |
| `fills` | broker_fill_id, broker_order_id, quantity, price, fee, filled_at | broker_fill_id unique |
| `snapshots` | run_id, captured_at, NAV, cash, positions, source | `(run_id, captured_at, source)` unique |
| `runs` | run_id, environment, account_hash, trading_date, signal_date, config_hash, status, notification_status, timestamps | run_id unique |

**상태값**

- 실행 상태는 `started`, `skipped`, `blocked`, `completed`, `failed`, `needs_review`이다.
- 주문 상태는 `planned`, `submitted`, `partially_filled`, `filled`, `cancel_pending`, `cancelled`, `unknown`이다.
- 체결·주문 행은 삭제하지 않고 상태와 조정 기록을 추가한다.
- 로그와 DB 백업은 Section 0.2의 보존 정책을 따른다.

## 6. 외부 인터페이스 (External Interfaces)

### 6.1 브로커 계약

```python
class Broker(Protocol):
    def account_snapshot(self) -> AccountSnapshot: ...
    def quote(self, symbol: str) -> Quote: ...
    def open_orders(self, since: datetime) -> Sequence[BrokerOrder]: ...
    def submit(self, request: OrderRequest) -> BrokerOrder: ...
    def cancel(self, broker_order_id: str) -> BrokerOrder: ...
    def order(self, broker_order_id: str) -> BrokerOrder: ...
    def fills(self, since: datetime) -> Sequence[Fill]: ...
```

- paper와 production은 동일 계약을 사용하고 base URL, 앱키, 계좌 allowlist를 분리한다.
- 인증 헤더와 민감한 응답 필드는 로그 전에 마스킹한다.
- 조회 API만 Section 0.2의 재시도 정책을 적용한다.
- 주문 API 오류와 타임아웃은 Section 3.5의 상태 조정 절차를 적용한다.

### 6.2 데이터 공급자 계약

```python
class MarketDataProvider(Protocol):
    def daily_bars(self, symbol: str, start: date, end: date) -> Sequence[DailyBar]: ...
    def regime_inputs(self, start: date, end: date) -> Sequence[RegimeInput]: ...

class SentimentProvider(Protocol):
    def daily_sentiment(self, symbol: str, session_date: date) -> SentimentObservation: ...
```

- 공급자별 심볼과 수정주가 의미를 canonical 모델로 변환한다.
- 호출 한도는 설정값으로 관리하고 공급자 오류와 한도 초과를 구분한다.

### 6.3 알림 계약

```python
class Notifier(Protocol):
    def send(self, message: Notification) -> DeliveryResult: ...
```

- 알림에 시크릿, 전체 계좌번호, 브로커 원문 응답을 포함하지 않는다.

## 7. 검증 및 출시 기준 (Validation & Rollout)

### 7.1 Phase 1 — 백테스트 승인

- Section 0.2의 기간과 walk-forward, holdout 정책을 적용한다.
- SPY buy-and-hold와 현금을 비교 기준으로 사용한다.
- Section 0.2의 거래비용 민감도별 결과를 모두 제시한다.
- 유니버스 등급 B를 기본 결과로 사용하고 등급 C를 민감도 결과로 함께 제시한다.
- 등급 B를 구성할 수 없으면 등급 C의 한계를 표시하고 Phase 2까지만 진행한다.
- 신규 매수 중단 정책과 노출 축소 정책을 동일 데이터와 비용 조건으로 평가한다.
- CAGR, MDD, Sharpe, Calmar, 회전율, 회복기간을 전체 기간과 주요 충격 구간에 보고한다.
- 미래 참조와 중복 일봉이 없고, 인접 파라미터에서 성과 방향이 유지되어야 한다.
- 승인된 전략 설정, 레짐 정책, 데이터 범위, 유니버스 등급과 해시를 실행 manifest로 보존한다.

### 7.2 Phase 2 — dry-run과 모의투자 승인

- KIS 모의투자 지원 범위를 확인한 후 주문 계약을 확정한다.
- Section 0.2의 모의투자 최소 기간을 연속 운영한다.
- 치명 오류가 없어야 한다.
- 모든 주문 의도는 브로커 주문 또는 명시적 미제출 사유와 연결되어야 한다.
- 주문 응답 유실과 강제 종료 후 중복 주문이 없어야 한다.
- 목표 수량과 최종 수량의 차이를 정수 반올림, 현금 여유분, 미체결로 설명할 수 있어야 한다.
- kill switch, stale data, 손실 한도, 부분 체결, 취소 실패를 실제 또는 장애 주입으로 검증한다.
- 유니버스 등급 B를 확보하지 못하면 Section 0.2의 보완 검증과 사용자 명시적 승인을 추가한다.

### 7.3 Phase 3 — 실계좌 단계적 출시

- Section 0.2의 실계좌 dry-run 기간을 먼저 수행한다.
- 사용자 승인 자본 상한으로 Section 0.2의 소액 검증 기간을 운영한다.
- 설명할 수 없는 잔고 차이 또는 치명 오류가 발생하면 신규 매수를 중단하고 Phase 2로 되돌린다.
- 자본 상한 확대는 사용자 승인과 설정 변경 기록으로만 수행한다.

### 7.4 필수 테스트

- 전략 결정성, 미래 참조, 결측·동점 처리 테스트
- 목표 비중 합계와 종목·슬리브 상한 테스트
- 정수 수량, 현금 부족, 수수료 여유분 테스트
- 손절, 손실 한도, stale data, kill switch 테스트
- 주문 멱등성과 주문 응답 유실 테스트
- 부분 체결과 취소 실패 테스트
- paper와 production 계좌 격리 테스트

- 작성일: 2026-07-23
- 리뷰어: Claude (Fable 5, Orca 교차검증 워커)
- 대상: main 브랜치 `205b201` 기준 전체 저장소 (읽기 전용 리뷰)

## 1. 요약

**전체 구조는 스펙 §0.3 불변식(fail-closed, 매도 우선, 멱등 주문 의도, 안전 게이트)을 충실히 따르는 잘 설계된 골격이다.** 전략 순수함수(백테스트·라이브 동일 엔진 `select_targets`), 2단계 리밸런스(매도→계좌 재조회→매수), permit 기반 안전 게이트, 킬스위치 파일, 설정 해시 등은 모범적이다. 단위 테스트 96건 전부 통과.

그러나 **실계좌 투입 전 반드시 고쳐야 할 High 이슈 5건**을 발견했다. 핵심은 (1) 취소된 주문이 로컬 원장에 "전량 체결"로 기록되는 브로커 상태 판정 결함, (2) `trading_date`=신호기준일 conflation으로 인한 멱등키 충돌(데이터 미갱신 시 거래가 조용히 누락), (3) 장중 `--refresh-data` 시 미완성 당일 바가 신호에 유입되는 look-ahead, (4) 프로세스 간 재실행 복구 미완성(크래시 후 재실행이 하드 크래시), (5) 분할(split) 발생 시 증분 수집이 팩터를 왜곡.

**수익 관점의 결론은 더 냉정하다.** grade-C 백테스트 자체가 편도 30bp에서 CAGR −2.2%로, **스펙 §7.1의 자체 승인 기준("가장 보수적인 비용 시나리오에서 현금보다 높은 기대수익")을 현재 미충족**이다. 또한 SPY 벤치마크가 데이터셋에 없어(벤치마크=현금뿐) "holdout MDD < SPY" 기준도 검증 불가 상태다. 실계좌 기대수익은 비용 우대율·생존편향 보정에 따라 **보수 −15% 안팎(운용 불가), 기본 세후 0~+4%(SPY 열위), 낙관(grade-B 후 회전율 절감 성공 시) 세후 +8~12%** 로 추정한다. 현행 파라미터(band=0, hysteresis=0, 연 회전율 ~107배)로 표준 수수료(0.25%) 운용은 확실한 손실 경로다.

## 2. 흐름 검증 결과 및 발견 이슈

### 2.1 파이프라인 연결 상태 (단계별)

| 단계 | 구현 | 연결 상태 | 비고 |
|---|---|---|---|
| 일일 수집 | `scripts/collect.py` (yfinance, upsert 멱등) | ✅ `daily_run --refresh-data`로 배선 | 갱신 실패 시 기존 DB로 진행(비치명) — 단, H2·H3와 결합하면 위험 |
| 품질 검증 | `build_market_data_snapshot` + `_build_clean_snapshot` | ✅ interior_gap·OHLC·중복·future_bar fail-closed | 캘린더가 prices 테이블 유래(공식 캘린더 아님) |
| daily_return | `NavHistory` (JSON, 전 거래일 미포함 조회) | ✅ 리스크 정책·SafetyContext 이중 배선, 일치 검증까지 있음 | "확정 NAV"가 아닌 실행시점 NAV 근사. 파일이 계좌/환경 미구분 |
| 주간 추천 | `scripts/recommend.py` | ✅ 동일 엔진(`signal`) 재사용, 리포트 전용(주문 무관) | 데이터 전달 결함 없음 — 자동매매와 독립 |
| 전략→리스크 | `signal` → `apply_risk_policy` | ✅ 보유종목 hysteresis 입력, 레짐 없으면 `regime_missing`으로 fail-closed | 손절 판정이 스펙(신호기준일 종가)과 달리 잔고 현재가 기준 |
| 주문 계획 | `plan_orders` (no-trade band, risk_reduced 우선) | ✅ Codex High 지적(리스크 축소 band 예외) 프로덕션에도 이식됨 | 매도 수량 환산에 buy_limit 사용(M2 참조) |
| 집행 | `execute_rebalance` (매도→재조회→매수, permit) | ✅ 매도 우선·현금 재계산 스펙 §3.1 준수 | |
| KIS 주문 | `kis_broker.py` (submit/cancel/status) | ⚠️ 체결 상태 판정 결함(H1), 호가 미사용(M1) | |
| 재실행 복구 | `_reconcile_open_orders` + `_submit_once` | ⚠️ 이월 방지·중복 제출 방지는 동작하나 복구 재실행은 크래시(H4) | |
| 킬스위치·런타임 | `configs/runtime.toml` + `KILL_SWITCH` 파일 + permit | ✅ production 3중 게이트(allow_real_orders/release_approved/allowlist) 견고 | runs 테이블 environment "paper" 하드코딩(M8) |
| 알림·감사 | `LogNotifier` + runs/orders/order_events | ⚠️ `RunRepository.finish` 미호출 — 모든 run이 영구 'started'(H4와 함께) | 텔레그램·외부 dead-man 채널 미구현(§8 미결) |

**결론: 정상 경로(happy path)는 단계 간 데이터·상태 전달에 결함 없이 이어진다.** 문제는 비정상 경로(재실행, 데이터 미갱신, 취소, 장중 수집)에 집중되어 있다.

### 2.2 High — 실계좌 전 필수 수정

**H1. 취소된 주문이 "전량 체결(filled)"로 기록된다** — `kis_broker.py:257-282`, `executor.py:61-69`
`KisBroker._status`는 미체결내역(nccs)에 없으면 무조건 `filled`로 간주한다. 그런데 취소된 주문도 미체결 목록에서 사라진다. 실행 흐름상 타임아웃 → `cancel()` → `await_terminal(cancelled)` 순서에서 `await_terminal`이 다시 `_status`를 호출하므로, **미체결 상태로 취소된 주문이 `filled + filled_quantity=전량`으로 로컬 DB(orders/order_events)에 기록**된다. 부분체결 수량도 체결내역(ccnl) 미조회로 정확하지 않다(코드 주석이 후속 과제로 자인). 결과: 로컬 원장이 오염되어 §7.2 "포지션 허용 차이" 정합성 판정이 불가능하고, H2와 결합하면 같은 신호일의 재실행에서 미체결분 재주문이 "이미 filled"로 오인되어 영구 스킵된다. `fills` 테이블은 스키마만 있고 미사용. **Phase 2(모의 20거래일) 시작 전 ccnl 기반 체결 조회로 반드시 교체해야 한다.**

**H2. trading_date=신호기준일 conflation → 멱등키 충돌로 거래가 조용히 누락된다** — `daily_run.py:95,157,247-248`, `planner.py:19-37`
스펙 §0.1은 주문 의도 ID를 "거래일 D" 기준으로 정의하지만, `daily_run`은 DB 최종 세션(`end` = 신호 기준일 D−1)을 `trading_date`이자 `signal_date`로 사용한다. `--max-staleness-days 4` 기본값에서 데이터 갱신이 하루 실패해도 실행은 "fresh"로 통과하는데, 이때 **어제와 동일한 intent_id가 생성되어 `_submit_once`가 전부 스킵(전일 상태 filled/cancelled → return None)하고, 실행은 "제출 0건"으로 정상 종료·통보**된다. 거래 누락이 성공처럼 보이는 최악의 침묵 실패다. run_id에는 타임스탬프가 붙지만 intent_id에는 실제 거래일이 없어 멱등성 경계가 "신호일"로 잘못 잡혀 있다. 수정: trading_date는 시장 캘린더 기반 실제 주문일 D로 분리하고, 같은 signal_date로 서로 다른 D에 두 번 주문하려는 경우를 명시적으로 감지·차단(또는 경보)해야 한다.

**H3. 장중 `--refresh-data` 실행 시 미완성 당일 바가 신호 기준일이 된다** — `collect.py:70-92`, `backtest.py:63-135`
yfinance `history(period="10d")`는 미국 장중에 호출하면 당일 진행 중(partial) 바를 포함한다. 스펙 §0.2 실행 시각 기본값이 **D 09:31 ET**이므로, 그 시각에 refresh하면 `MAX(session_date)` = 오늘이 되고, 미완성 바는 OHLC 관계가 유효해 `cap_final`(불량 최종 세션 제외)로 걸러지지 않는다. 결과: **"신호 데이터는 신호 기준일까지 확정된 일봉만 포함한다"(§0.3) 불변식 위반** — 모멘텀·SMA·손절·레짐이 장중 스냅샷으로 계산된다. H2와 정반대 방향의 오류라 staleness 검사로도 안 잡힌다. 수정: 수집 시 현재 ET 시각이 16:00 이전이면 당일 행을 드롭하거나, 스냅샷 구성에서 `end < 오늘(ET)`을 강제.

**H4. 프로세스 간 재실행 복구가 미완성 — 크래시 후 재실행이 하드 크래시하고 run이 영구 'started'로 남는다** — `kis_broker.py:188-195`, `executor.py:44-48`, `daily_run.py`
`find_by_intent`는 프로세스 내 메모리 dict만 조회한다(주석이 "프로세스 간 멱등성은 후속"으로 자인). 제출 후 크래시 → 재실행 시: DB에 intent가 남고(비터미널 status) → reserve 실패 → 브로커 조회 결과 None → `ExecutionUncertain` 발생. 그런데 `daily_run._execute_live`는 `SafetyBlocked`와 `KisApiError`만 잡으므로 **`ExecutionUncertain`은 처리되지 않은 예외로 전파**되어 traceback 종료한다. 중복 주문은 없으니 fail-closed 자체는 지켜지지만, (a) 복구 절차(§3.5 "기존 주문 상태를 추적")가 없고 (b) `RunRepository.finish`가 코드 전체에서 한 번도 호출되지 않아 **모든 run이 영구 'started'** — §3.7 dead-man 감시("성공 또는 명시적 휴장 기록")의 전제가 성립하지 않으며, 경보 알림도 없다. 수정: orders 테이블의 broker_order_id로 브로커 대조(보조 대조키 §3.5), ExecutionUncertain → needs_review 기록 + critical 알림, try/finally로 finish 보장.

**H5. 분할(split) 발생 시 증분 수집이 팩터를 왜곡한다 — 자동 감지 없음** — `collect.py:77-79`
일일 운영은 `period="10d"` 증분 수집인데, 분할이 발생하면 과거 행의 split_adjusted 가격이 재작성되지 않아 최근 10일과 과거 이력의 수정주가 스케일이 어긋난다(주석이 "전체 재수집 필요"로 자인하나 감지 로직이 없다). 2:1 분할이면 모멘텀이 −50%로 계산되어 해당 종목이 조용히 탈락하고, 200SMA·MACD도 왜곡된다. 보유 종목이면 부당 청산 주문까지 나갈 수 있다. 수정: 수집 시 `Stock Splits != 0` 행 발견 즉시 해당 심볼 전체 이력 재수집(또는 실행 차단+경보).

### 2.3 Medium

**M1. 호가 미사용 — last를 bid/ask로 대체** (`kis_broker.py:148-155`). 스펙 §3.5 "기준 가격은 최우선 호가, 호가 없으면 주문 생성 금지"와 어긋난다. 지정가 ±0.3%가 last 기준이라 스프레드가 넓은 종목은 매수는 공격적, 매도는 미체결 위험. 백테스트(시가 체결)와의 괴리가 슬리피지로 누적된다(§3 수익 추정에 반영).

**M2. 레짐/일손실 축소 기간에 1주 단위 'shaving' 매도가 반복 발생.** `min(target, current_weight)`로 목표=현재 비중을 만들어도, planner가 목표 수량을 `buy_limit(ask×1.003)`로 환산하므로 floor에서 현재 수량보다 1주 적게 나와 매 실행 소량 매도가 나간다. 이 종목들은 `risk_reduced_symbols`로 분류되어 no-trade band 예외라 band로도 못 막는다. 레짐 장기화 시(2022년형 52세션+) 불필요한 회전·비용. 매도 수량 환산은 현재가(또는 sell_limit) 기준으로 분리 권장.

**M3. 계좌 스냅샷 가정이 미검증.** NAV=KRW 총자산/환율(통합증거금 가정), 보유는 `inquire_balance(NASDAQ)` 단일 조회. NYSE/AMEX 보유가 응답에 안 실리면 보유 누락→해당 종목 매도 불가+현금 과대평가→과매수. 코드 주석 스스로 "실보유 발생 시 재확인" 전제이므로, 모의 첫 체결 후 즉시 검증 항목으로 못 박아야 한다. 현금을 USD 예수금이 아닌 잔차(NAV−보유평가)로 구해 자동환전 불가 계좌에서는 주문 거부 가능성.

**M4. `_reconcile_open_orders`가 계좌의 모든 미체결을 무차별 취소** (`daily_run.py:191-210`). 사용자가 수동으로 낸 주문도 함께 취소된다. 시스템 계좌 전용이라는 전제가 스펙·설정 어디에도 명시돼 있지 않다.

**M5. EQUITY_CAP 정책의 초과분 차감이 음수 비중을 만들 수 있다** (`regime.py:132-136`). `adjusted[first_symbol] -= excess`에서 첫 심볼 비중 < excess면 음수가 되어 `plan_orders`가 ValueError로 크래시. 실험 정책이지만 `--policy equity_cap`로 노출되어 있다.

**M6. SPY 벤치마크 부재.** 데이터셋에 SPY가 없어(`sqlite_provider.py` 주석) `gradeC_full_evaluate.json`의 벤치마크가 현금뿐이다. §7.1 승인 기준 "holdout 비용차감 MDD < SPY"를 **검증할 수 없는 상태**. 수집 유니버스에 SPY 추가가 선행돼야 한다.

**M7. 고변동 슬리브가 라이브에 미배선.** `daily_run`은 `high_volatility_symbols`를 전달하지 않아 88%만 배분되고 10%는 상시 현금이다. 백테스트(슬리브 포함)와 목표 종목·수량이 달라져 §7.2 "백테스터·모의투자 목표 일치" 기준이 성립하지 않는다. 의도적이면 스펙에 명시 필요.

**M8. runs.environment가 "paper" 하드코딩** (`daily_run.py:245`). runtime.toml이 production이어도 감사 기록은 paper로 남는다. 감사성(§4) 훼손.

### 2.4 Low / 관찰

- **VIX·WTI 캘린더 불일치 → spurious 레짐 발동**: 한쪽 지표만 있는 날은 invalid→active(fail-closed)가 되고 해제에 3연속 유효일이 필요해 며칠씩 매수가 차단된다. 다만 백테스트도 같은 로직·데이터라 성과 수치에는 이미 반영되어 있어 정합성 문제는 아니다(비용일 뿐).
- 손절 판정이 잔고 스냅샷의 현재가(now_pric2) 기준 — 스펙(신호 기준일 종가 −10%)과 다르다. 보수적 방향이지만 스펙과 문서화 필요.
- `date.today()`(KST) 기준 staleness 판정 — ET 세션과 어긋날 수 있으나 4일 여유로 실害는 낮다(단 H2 참조: 여유가 오히려 침묵 실패를 허용).
- 토큰 캐시(`data/kis_token.json`)·nav_history 파일 권한 미제한(§4 파일 권한 NFR), nav_history가 계좌/환경 미구분.
- `_get/_post`가 HTTP 상태코드 미검사(비JSON 5xx 응답 시 json 파싱 예외), 401 토큰 만료 시 재발급 재시도 없음. 조회 재시도도 rate-limit 1회뿐(스펙 §0.2: 최대 3회 지수 백오프).
- `within_us_market_hours`는 미국 공휴일·조기폐장 미반영(문서화됨, 브로커 fail-closed로 완화). 조기폐장일 13:00 이후 제출분은 주문 거부에 의존.
- 주간거래(daytime) 경로는 kis.py에 구현됐지만 KisBroker.submit이 사용하지 않음(정규장 전용) — 유동성 얇은 주간거래 미사용은 현재로선 올바른 선택.
- 텔레그램 Notifier·외부 dead-man 채널(§8 미결 7번) 미구현 — LogNotifier(stdout+파일)만 존재.

### 2.5 엣지케이스 매트릭스 (요청 항목별)

| 엣지케이스 | 판정 | 근거 |
|---|---|---|
| 장 휴일 | ⚠️ 부분 | 캘린더가 prices 유래라 "휴장 인지"는 데이터 부재로 간접 처리. 주말은 `within_us_market_hours`로 차단, 공휴일은 브로커 거부 의존. 스펙 §3.1 "휴장일 = 주문 없이 성공 종료+사유 기록"의 명시 경로 없음 |
| 부분체결 | ❌ 결함 | H1 — nccs 잔량으로 추정만 하고 ccnl 미사용, 취소=filled 오기록 |
| API 실패 | ✅/⚠️ | 조회 실패·장종료는 fail-closed(exit 2/3). 단 rate-limit 외 재시도 없음, HTTP 상태 미검사, 제출 타임아웃 대조는 골격만(H4) |
| 중복 실행 | ⚠️ 부분 | intent_id 멱등으로 같은 날 중복 제출은 방지. 그러나 프로세스 락(§3.1 단계 1) 미구현 — 동시 실행 두 프로세스는 reserve 경합으로 대체될 뿐, stale lock 처리 없음. H2의 침묵 스킵도 중복 실행 처리의 부작용 |
| 레짐 전환 시점 | ✅/⚠️ | 진입 즉시·해제 3연속 확인, 비대칭 exit 임계는 bit-identical 기본값으로 대기 중(설계 양호). M2 shaving 매도와 spurious 발동(캘린더 불일치)이 잔여 이슈 |

## 3. 실계좌 기대수익 추정

### 3.1 근거 수치 (grade-C, 494종목, 2016-01~2026-07, BUY_BLOCK, `gradeC_full_evaluate.json`)

| 편도 비용 | 전체기간 CAGR | MDD | 비고 |
|---|---|---|---|
| 5bp | +33.0% | −43.3% | |
| 15bp | +17.8% | −55.0% | holdout(2024-26) +37.0%/−24.1%, walk-forward: −35.3% / +8.8% / +170%대 |
| 30bp | **−2.2%** | −78.7% | **§7.1 승인 기준(보수 비용에서 현금 초과) 미충족** |

- 연환산 회전율 ≈ 107배(turnover 1,120/10.55년) → **편도 1bp당 연 약 1.4%p의 CAGR 민감도** (표에서 25bp 차이에 35.2%p, 복리 효과 포함).
- 백테스트 체결 가정: 다음날 **시가 전량 체결**(부분체결·미체결·호가 스프레드 없음), 배당 현금 반영, grade-C(현재 생존 종목만) 유니버스.
- 기존 리뷰 문서가 명시한 한계: 생존편향으로 hysteresis 튜닝 시 CAGR 18%→49% 요동(편향 증폭 증거), 단일 realization, walk-forward 구간 간 분산 극단적(−35%~+170%).

### 3.2 실전 손실 요인 정량화

| 요인 | 추정치(편도 환산 또는 연 %p) | 근거 |
|---|---|---|
| KIS 해외주식 수수료 | 표준 25bp / 우대·이벤트 시 7~10bp | 공시 수수료율. **이 한 항목이 생사를 가른다** |
| 규제 비용(SEC fee+TAF, 매도측) | 평균 편도 ~0.2bp | 매도금액의 ~0.28bp+건당 TAF |
| 슬리피지 | 5~15bp(대형주), 10~25bp(모멘텀 상위·개장 직후) | 백테스트는 시가 무마찰 체결 가정. 실제는 last±0.3% 지정가(M1: 호가 미사용)로 개장 변동 구간 체결. 미체결→취소→다음날 재계산의 기회비용 별도 |
| 미체결/부분체결 drift | 연 −0.5~−2%p | 10분 유효 지정가라 급등 추격 매수는 체결 실패가 구조적. 모멘텀 전략에서 미체결 편향은 "오르는 걸 못 삼"으로 귀결(불리한 방향) |
| 환전 | 왕복 0.1%(우대 90%+)~1%(창구) — 1회성 | 연 수익률 영향 미미. 통합증거금 자동환전 시 환가료 별도 |
| 양도소득세 | 실현이익의 22%(기본공제 연 250만원) | 연 회전율 107배 → 이익이 사실상 전액 매년 실현됨 → 이연 효과 0. 세후 ≈ 세전 양수익 × 0.78 |
| 생존편향(grade-C) | CAGR −5~−15%p (구간 추정) | 상위 5종목 집중 모멘텀 + "2026년 생존 종목" 유니버스 조합은 편향 민감도가 가장 큰 유형. 자체 문서(회전율 개정안 §4)도 정량 분리 불가·grade-B 필요를 인정 |
| 주간거래 유동성 | 해당 없음(현재 정규장만 제출) | 주간거래 세션 사용 시 스프레드 20~50bp — 사용 금지 유지 권장 |

실효 편도 비용 c ≈ **수수료 + 슬리피지**: 우대 수수료 확보 시 c ≈ 17~25bp, 표준 수수료 시 c ≈ 30~50bp.

### 3.3 시나리오 (실계좌, 현행 파라미터: band=0, hysteresis=0, BUY_BLOCK)

비용 보간(1bp ≈ 1.4%p) 후 생존편향 haircut과 세금을 순차 적용했다. 모두 grade-C 단일 realization 기반이므로 **점추정이 아니라 구간**으로 읽어야 한다.

| 시나리오 | 가정 | 세전 CAGR | 세후 CAGR | 예상 MDD |
|---|---|---|---|---|
| **보수** | 표준 수수료(25bp)+슬리피지 15bp ⇒ c≈40bp, 편향 haircut −10%p | **−15% ± 5%p** | 동일(손실) | −70% 이상 |
| **기본** | 우대 수수료(8bp)+슬리피지 12bp ⇒ c≈20bp ⇒ grade-C 외삽 +11%, 편향 haircut −8%p | **+1~+5%** | **0~+4%** | −50~−60% |
| **낙관** | grade-B 확보 + hysteresis/band 튜닝으로 회전율 50% 절감(문서 §3 레버 B 근거: 1120→550) ⇒ 실효 비용 부담 절반, 편향 haircut −5%p | **+12~+18%** | **+9~+14%** | −40~−55% |

**해석:**
- **보수 시나리오는 운용 불가가 아니라 확정적 자본 잠식이다.** 30bp에서 grade-C(편향으로 부풀려진) 수치조차 −2.2%인데, 실전은 여기서 더 나빠질 수만 있다. 수수료 우대를 확보하지 못하면 시작하면 안 된다.
- **기본 시나리오조차 SPY buy-and-hold(장기 ~10%, MDD −25% 내외)에 수익·MDD 모두 열위다.** 세후 0~4%를 위해 −55% 낙폭을 감내하는 구조. holdout +37%는 2024-26 강세장 + 생존편향의 산물로 기대치로 삼으면 안 되고, walk-forward 1(2021-22)의 −35%가 실제로 겪게 될 하방의 모습이다.
- **낙관 시나리오는 3가지 전제가 모두 성립해야 한다**: (a) grade-B PIT 데이터 확보, (b) hysteresis 튜닝이 편향이 아닌 실제 엣지로 확인, (c) 우대 수수료. 이 중 (b)는 현재 검증 불가(문서 스스로 인정)이므로 계획이지 기대값이 아니다.
- 부호를 가르는 두 변수는 **수수료 우대율**과 **회전율**이며 전략 알파가 아니다. 연 107배 회전율에서는 비용이 수익 구조를 지배한다(연 비용 ≈ 107 × c).

### 3.4 추정의 한계

생존편향 haircut(−5~−15%p)은 grade-B 대조군이 없어 문헌·구조적 추론(집중 모멘텀 + 현재 구성원 유니버스) 기반의 판단 구간이다. 슬리피지도 실체결 데이터가 없어(모의 왕복 미완) 사전 추정이다. Phase 2 모의 20거래일에서 **체결가 vs 백테스트 시가**의 실측 분포로 이 표를 교정하는 것이 다음 단계다.

## 4. 권고사항

### 즉시 (Phase 2 모의투자 시작 전 — 차단 조건)

1. **[H1] ccnl(주문체결내역) 기반 체결 원장 구현**: 취소≠체결 구분, 부분체결 수량 정확화, `fills` 테이블 기록. 이것 없이는 §7.2 정합성 검증 자체가 불가능하다.
2. **[H2] trading_date를 캘린더 기반 실제 주문일 D로 분리**하고, 동일 signal_date의 이틀 연속 사용을 감지해 차단·경보. "제출 0건"과 "멱등 스킵"을 알림에서 구분 표기.
3. **[H3] 장중 수집 가드**: ET 16:00 이전 수집 시 당일 바 드롭(collect 또는 snapshot 구성 시 `end < today(ET)` 강제).
4. **[H4] ExecutionUncertain 처리 경로**: needs_review 기록 + critical 알림 + orders.broker_order_id 기반 브로커 대조 복구. `RunRepository.finish`를 try/finally로 전 경로 보장.
5. **[H5] 분할 감지**: 수집 중 `Stock Splits != 0` 발견 시 해당 심볼 전체 재수집(자동) 또는 실행 차단.

### 단기 (모의투자 기간 중)

6. **[M6] SPY 수집 추가** — §7.1 승인 기준 검증의 전제.
7. **[M3] 첫 실체결 후 잔고 응답 필드·거래소 커버리지 실측 검증** (NYSE 보유 포함 여부, 현금 vs USD 예수금).
8. **[M1] 호가 엔드포인트 적용** + 체결가 vs 백테스트 시가 슬리피지 실측 로깅(§3.4 교정용).
9. **[M2] 매도 수량 환산 기준가 분리**, **[M5] equity_cap 음수 비중 수정**, **[M7] 고변동 슬리브 배선 여부 스펙 확정**, **[M8] environment 하드코딩 제거**.
10. 프로세스 락(§3.1 단계 1)·휴장일 명시 처리·텔레그램/외부 dead-man 채널(§8 미결 7번) 구현.

### 전략·수익성 (Phase 1 재승인 필요)

11. **§7.1 승인 기준 미충족 상태를 공식화하라.** 30bp에서 CAGR −2.2%는 "가장 보수적인 비용 시나리오에서 현금 초과" 기준 탈락이다. 현행 파라미터로는 Phase 3 진행 근거가 없으며, 회전율 절감(grade-B + hysteresis/band 튜닝)이 성공해야만 기준 재평가가 가능하다.
12. **수수료 우대 확보를 운용 개시의 하드 게이트로 설정**하라(예: 편도 총비용 실측 ≤ 20bp 확인 전 실계좌 금지). §3.3의 민감도(1bp ≈ 연 1.4%p)를 근거로.
13. grade-B PIT 데이터 확보를 최우선 투자로 — 파라미터 튜닝, 생존편향 보정, 승인 기준 재평가가 전부 여기 걸려 있다.
14. 실계좌 소액 검증(Phase 3) 자본 상한 결정 시 **기본 시나리오(세후 0~4%, MDD −55%)를 기준으로 감내 가능한 절대 손실액**으로 역산할 것을 권장한다.
